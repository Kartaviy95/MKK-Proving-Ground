Дизайн проекта
MKK Proving Ground

Техническое имя проекта: mkk_ptg
Префикс сборки: ptg
Назначение: современный модульный полигон для Arma 3 с отдельным стартовым меню функций, серверным спавном техники, каталогом по динамическим фракциям/нациям, телепортом, мишенями и системой слежения за пулей или снарядом.

1. Общая идея проекта

MKK Proving Ground — это отдельный модульный полигон для быстрого теста техники, вооружения, боеприпасов и сценариев боя в Arma 3.

Он должен заменить старый перегруженный полигон на более чистый, понятный и расширяемый инструмент.

Главная идея проекта — не открывать игроку сразу огромный список техники, а сначала показать центральное меню функций полигона.

Игрок должен сам выбрать, что ему нужно:

создать технику;
телепортироваться;
запустить тест пробития техники;
создать мишень;
включить или отключить слежение за пулей/снарядом;
включить или отключить линию траектории пули;
очистить полигон;
обновить каталог;
перейти в будущие модули.

Каталог техники является важной частью проекта, но он не должен быть первым экраном. Он открывается только после выбора функции “Создание техники”.

2. Основная философия дизайна

Проект строится по четырем принципам.

1. Простота

Игрок не должен разбираться в технических деталях, чтобы воспользоваться полигоном.
Ключевые действия должны быть доступны за 2–3 клика.

2. Модульность

Каждая крупная функция живет отдельным модулем:

ядро;
общие функции;
каталог;
интерфейс;
спавн;
мишени;
слежение;
ACE-интеграция.

Новые функции добавляются как отдельные экраны или модули, а не встраиваются хаотично в одно окно.

3. Современный внешний вид

Интерфейс должен выглядеть как рабочий военный технический терминал.
Он не должен ощущаться как временное debug-меню.

4. Локализация

Весь видимый текст аддона должен находиться в stringtable.
Минимально поддерживаются два языка:

English;
Russian.

Все кнопки, подсказки, ACE actions, HUD tracking и подписи интерфейса должны брать текст из stringtable.

3. Роль проекта в сборке

Проект используется как:

тестовый полигон для техники;
среда для проверки вооружения;
площадка для отладки поведения боеприпасов;
инструмент быстрого вызова объектов;
инструмент быстрой телепортации;
средство создания статичных и пехотных целей;
база для будущих расширений: журнал пробитий, сценарии испытаний, пресеты, расширенные настройки tracking.

4. Визуальная концепция

Визуально проект должен ощущаться как военный технический терминал.

Общий стиль:

темный графитовый фон;
светлый контрастный текст;
аккуратные рамки;
минимализм без перегруза;
четкое деление на функциональные зоны;
приглушенный синий/бирюзовый акцент;
красный акцент только для опасных действий вроде очистки полигона.

Игрок должен чувствовать, что это:

не чит-меню;
не временная отладочная панель;
а штатный инструмент полигона.

5. Цветовая схема

Базовые цвета:

фон: темно-серый / графитовый;
панели: чуть светлее фона;
текст: белый или светло-серый;
акцент: темно-синий / голубой / приглушенный бирюзовый;
предупреждения: желтый;
опасные действия: красный;
успех / подтверждение: зеленый.

Цвета не должны быть декоративными сами по себе. Их задача — быстро подсказать назначение действия.

6. Общая структура интерфейса

Главное окно открывается как стартовая панель функций.

Стартовый экран содержит:

название проекта;
заголовок текущего раздела;
кнопку “Создание техники”;
кнопку “Телепорт”;
кнопку “Слежение за пулей”;
кнопку “Статичная мишень”;
кнопку “Пехотная цель”;
кнопку “Тест пробития”;
кнопку “Линия траектории пули”;
кнопку “Маркеры снарядов на карте”;
кнопку “Класс ammo на маркере”;
кнопку “Очистить полигон”;
кнопку “Обновить каталог”;
кнопку “Закрыть”.

Функциональные кнопки стартового экрана располагаются ровной симметричной сеткой и используют одинаковый размер.

Каталог техники скрыт при первом открытии окна.
Он появляется только после нажатия “Создание техники”.

7. Логика пользовательского опыта

Основной UX:

Открыл полигон → выбрал функцию → выполнил действие → вернулся назад или закрыл окно.

Для техники:

Открыл полигон → Создание техники → нашел технику → увидел карточку → задал дистанцию и угол → заспавнил → протестировал → очистил полигон.

Такой подход сохраняет главное окно чистым и не заставляет игрока каждый раз видеть каталог, если ему нужен телепорт, мишени, tracking, линия траектории или тест пробития.

8. Экран создания техники

Экран создания техники состоит из пяти зон.

Верхняя зона:

название проекта;
текущий раздел “Создание техники”;
счетчик найденных объектов.

Левая колонка:

поиск;
сторона;
фракция;
нация;
тип техники;
дистанция спавна в метрах;
угол разворота в градусах.

Центральная область:

список найденной техники;
быстрый выбор;
прокрутка списка.

Правая колонка:

изображение техники;
карточка выбранной техники.

Нижняя панель:

создать пустую технику;
создать технику с экипажем;
создать статичную мишень;
создать пехотную цель;
очистить полигон;
обновить каталог;
назад.

Кнопки нижней панели должны быть выровнены ровно и использовать единый размер, чтобы экран не выглядел как набор временных debug-элементов.

9. Каталог техники

Каталог строится динамически из CfgVehicles.

Не допускается жестко прописывать конкретные фракции из отдельных модов.
Проект должен корректно видеть технику из кастомных модов, если она зарегистрирована в конфиге Arma 3.

Для каждой единицы техники сохраняются:

classname;
displayName техники;
сторона;
classname фракции;
displayName фракции;
нация;
тип техники;
класс экипажа;
editorPreview;
picture;
editorSubcategory;
displayName группы / подкатегории;
источник мода.

Названия техники, фракций и групп должны проходить через локализацию:

если displayName содержит обычный текст, он используется напрямую;
если displayName содержит $STR_..., он должен быть преобразован через localize;
если значение отсутствует, используется безопасный fallback.

10. Фильтрация каталога

Фильтры должны работать по стабильным техническим значениям, а отображать локализованные названия.

Например:

фильтр фракции хранит classname фракции;
в UI показывается displayName фракции;
фильтр типа хранит ключ типа;
в UI показывается локализованное название типа.

Такой подход нужен, чтобы фильтрация не ломалась при смене языка игры.

Уровни фильтрации:

сторона;
фракция;
нация;
тип техники;
поиск по названию и classname.

11. Типы техники

Типы техники определяются автоматически по наследованию classname.

Поддерживаемые типы:

Tank;
IFV;
APC;
Car;
Truck;
Static;
Helicopter;
Plane;
Boat;
UAV;
Other.

В UI типы должны отображаться через stringtable:

Tank → Танк;
IFV → БМП;
APC → БТР;
Car → Автомобиль;
Truck → Грузовик;
Static → Статичное;
Helicopter → Вертолет;
Plane → Самолет;
Boat → Лодка;
UAV → БПЛА;
Other → Другое.

12. Карточка выбранной техники

Карточка должна показывать:

название;
classname;
изображение;
сторону;
фракцию;
нацию;
тип;
группу / editorSubcategory;
класс экипажа;
источник мода.

Все подписи карточки должны быть локализованы через stringtable.

Основная задача карточки:

дать игроку быстро понять, что это за техника, из какой она фракции/группы, из какого источника пришла и стоит ли ее спавнить.

13. Визуальный вид техники

Для первой версии визуальный вид работает по простой и надежной схеме:

editorPreview;
picture;
пустое значение или будущая заглушка.

Если у техники нет изображения или путь сломан, интерфейс не должен ломаться.

14. Спавн техники

Спавн техники должен выполняться по серверной схеме.

Клиент:

выбирает classname;
задает дистанцию;
задает угол разворота;
выбирает пустой спавн или спавн с экипажем;
отправляет запрос на сервер.

Сервер:

проверяет входные параметры;
создает технику на заданной дистанции перед игроком;
поворачивает технику с учетом угла разворота;
при необходимости создает экипаж;
регистрирует созданный объект в реестре полигона.

Дистанция ограничивается настройкой максимальной дистанции.

15. Мишени

Проект поддерживает:

статичную мишень;
пехотную цель.

Создание мишеней выполняется на сервере.
Все созданные мишени должны регистрироваться в реестре полигона, чтобы их можно было удалить через очистку.

16. Очистка полигона

Очистка должна удалять:

созданную технику;
созданные мишени;
прочие зарегистрированные объекты полигона.

Удаление выполняется сервером.

17. Телепорт

Телепорт — отдельная функция стартового меню.

Базовая логика:

игрок нажимает кнопку “Телепорт”;
функциональное меню скрывается или закрывается, чтобы не перекрывать карту;
открывается карта;
следующий клик по карте переносит игрока или его текущую машину в выбранную точку;
карта закрывается;
игрок получает локализованное уведомление.

В будущем телепорт можно расширить ограничениями доступа, выбором высоты, сохраненными точками и телепортом группы.

18. Система слежения за пулей/снарядом

Tracking является отдельной функцией полигона.

Стартовое меню должно позволять быстро включать или отключать tracking.

Система слежения должна быть:

понятной;
визуально чистой;
не перегруженной;
не мешающей тесту.

Режимы:

Simple;
Tactical;
Cinematic.

HUD tracking показывает:

тип боеприпаса;
время полета;
дистанцию;
текущий режим tracking.

Текст HUD должен быть локализован через stringtable.

19. Линия траектории пули

Линия траектории пули — отдельный toggle стартового меню.

Назначение:

визуально показать путь projectile после выстрела;
оставить линию на короткое время после исчезновения projectile;
помогать анализировать угол, падение и направление полета.

Техническая идея:

при FiredMan event handler сохраняются точки projectile;
Draw3D рисует линию между сохраненными точками;
линии живут ограниченное время;
выключение режима очищает сохраненные линии.

Линия траектории не должна обязательно включать cinematic tracking-камеру.
Это отдельный режим анализа.

20. Маркеры снарядов на карте

Маркеры снарядов на карте — отдельный режим анализа projectile.

Назначение:

после выстрела создать красную точку на карте;
перемещать точку за projectile во время полета;
оставить точку в последней позиции после попадания или исчезновения projectile;
по желанию показывать рядом classname ammo.

Управление:

кнопка “Маркеры снарядов на карте” включает или выключает сам режим;
кнопка “Класс ammo на маркере” включает или выключает подпись classname ammo.

Техническая идея:

при FiredMan event handler или тестовом projectile запускается локальная запись позиции projectile;
создается локальный красный marker типа mil_dot;
маркер обновляется до исчезновения projectile или таймаута;
после завершения маркер остается в последней известной позиции;
старые маркеры ограничиваются по количеству.

21. Тест пробития техники

Тест пробития — отдельный модуль главного меню.

Назначение:

создать тестовую технику;
посадить внутрь экипаж;
выбрать боеприпас;
навести курсор на нужную часть техники;
запустить тестовый снаряд в выбранную точку;
показать отчет по повреждениям техники и экипажа.

Экран теста пробития содержит:

список техники из общего каталога;
поиск по технике;
список боеприпасов из CfgAmmo;
поиск по боеприпасам;
кнопку создания тестовой цели;
кнопку orbit-камеры;
кнопку тестового выстрела;
кнопку сброса теста;
панель отчета повреждений;
компактный HUD отчета справа сверху во время режима камеры.

Для первой версии допустим MVP-режим:

снаряд выпускается в центр цели;
после попадания собираются damage, hitpoints и состояние экипажа;
отчет показывается в окне модуля.

Уточненная логика режима камеры:

тестовая техника создается на безопасной дистанции от игрока;
дистанция цели и дистанция создания тестового projectile задаются через внутреннюю конфигурацию аддона;
при включении orbit-камеры большое меню закрывается, чтобы не перекрывать цель;
выбранный ammo classname сохраняется перед закрытием меню;
orbit-камера управляется вручную: мышь вращает камеру, колесо меняет дистанцию;
выход из orbit-камеры выполняется по ESC;
левый клик мышью в режиме камеры определяет точку попадания через raycast по экрану;
снаряд создается на безопасной дистанции и направляется в выбранную точку;
отчет повреждений обновляется в компактном HUD справа сверху.

Список боеприпасов:

строится из CfgAmmo;
показывает только боеприпасы с projectile simulation;
не использует scope как обязательный фильтр, потому что у CfgAmmo он ненадежен;
исключает классы, у которых classname содержит base;
исключает классы без урона.

Будущие расширения:

настройка дистанции выстрела;
настройка угла встречи;
настройка скорости projectile;
сравнение нескольких боеприпасов;
экспорт отчета;
интеграция с ACE medical для статуса экипажа;
опциональная интеграция с Virtual Garage для кастомизации внешнего вида цели.

Virtual Garage можно использовать как дополнительный режим выбора/кастомизации техники, но основной тест пробития должен оставаться собственным модулем проекта.

22. Доступ к проекту

Открытие полигона происходит через:

горячую клавишу Home;
ACE self action;
ACE action на терминалах.

Так как аддон предназначен для личного пользования, доступ к полигону не ограничивается whitelist, admin access или внешними режимами допуска.

Все видимые строки ACE и keybind должны быть локализованы.

23. Внутренняя конфигурация

Настройки проекта хранятся внутри аддона, а не во внешнем меню настроек.

Базовые значения редактируются напрямую в файлах инициализации модулей и используются как личные дефолты.

Основные группы внутренних параметров:

Tracking;
Spawn;
Penetration.

Внешние блокировки, whitelist и отдельные пользовательские меню настроек для этой версии не нужны.

24. Локализация

Основной файл локализации:

addons/main/stringtable.xml

Минимальные языки:

English;
Russian.

Правила:

в конфиге UI использовать text = "$STR_MKK_PTG_...";
в SQF использовать localize "STR_MKK_PTG_...";
не хранить пользовательский текст напрямую в action-строках;
для динамического текста использовать format [localize "...", ...];
для displayName из чужих модов использовать общий helper локализации;
если строка из чужого мода не локализуется, показывать безопасный fallback.

25. Внутренняя архитектура

Фактические модули проекта:

ptg_main;
ptg_common;
ptg_catalog;
ptg_ui;
ptg_spawn;
ptg_targets;
ptg_tracking;
ptg_penetration;
ptg_ace.

Назначение модулей:

main — внутренняя конфигурация, keybind, открытие UI;
common — общие helpers, безопасное чтение конфигов, локализация строк;
catalog — сбор, фильтрация и описание техники;
ui — диалог, стартовое меню, экран техники, обработчики кнопок;
spawn — серверный спавн и реестр созданных объектов;
targets — создание и удаление целей;
tracking — слежение за projectile;
penetration — тест пробития техники, тестовый projectile, отчет damage/hitpoints/crew;
ace — ACE self actions и terminal actions.

26. Эмоциональное ощущение от проекта

Проект должен создавать ощущение:

аккуратности;
контроля;
технической надежности;
профессионального инструмента.

Игрок не должен чувствовать, что работает с кучей старых скриптов.
Он должен чувствовать, что у него в руках единый современный полигонный комплекс.

27. Что особенно важно сохранить в дальнейшем

1. Чистоту интерфейса

Не превращать стартовое окно в свалку функций.
Новые функции добавлять как отдельные модули или отдельные экраны.

2. Понятность каталога

Техника должна находиться легко.
Фракции и группы должны показываться нормальными названиями из конфигов и stringtable.

3. Стабильность сети

Спавн, удаление и создание объектов выполнять по правильной серверной схеме.

4. Локализацию

Не добавлять новый пользовательский текст напрямую в SQF или UI config.
Любой новый текст должен попадать в stringtable сразу.

5. Совместимость с кастомными модами

Не привязываться к конкретным фракциям или модам.
Каталог должен работать с любыми корректно описанными CfgVehicles, CfgFactionClasses и CfgEditorSubcategories.
