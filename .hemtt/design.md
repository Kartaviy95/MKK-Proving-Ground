Дизайн проекта
MKK Proving Ground

Техническое имя: mkk_ptg
Назначение: современный полигон для Arma 3 с удобным интерфейсом, серверным спавном техники, каталогом по фракциям/нациям и новой системой слежки за пулей или снарядом.

1. Общая идея проекта

MKK Proving Ground — это отдельный модульный полигон, который нужен для быстрого теста техники, вооружения и сценариев боя в Arma 3.
Он должен заменить старый перегруженный и устаревший полигон на новый, более чистый, удобный и визуально понятный инструмент.

Главная идея проекта — не просто открыть список техники и заспавнить объект, а дать игроку понятную рабочую среду, где можно:

быстро найти нужную технику;
увидеть ее визуально до спавна;
понять, к какой стороне, фракции и нации она относится;
заспавнить ее в нужной конфигурации;
сразу проверить оружие, броню или поведение;
включить слежку за пулей/снарядом;
очистить полигон без мусора и лишних действий.
2. Основная философия дизайна

Проект должен быть построен по трем принципам.

1. Простота

Игрок не должен разбираться в технических деталях, чтобы воспользоваться полигоном.
Все ключевые действия должны быть доступны за 2–3 клика.

2. Модульность

Полигон должен легко расширяться.
Новые функции добавляются отдельными модулями, а не переписывают старое ядро.

3. Современный внешний вид

Интерфейс должен выглядеть аккуратно, строго и понятно.
Не как старое debug-меню, а как полноценный инструмент тестирования.

3. Роль проекта в сборке

Проект используется как:

тестовый полигон для техники;
среда для проверки вооружения;
площадка для отладки поведения боеприпасов;
инструмент быстрого вызова нужных объектов;
база для будущих расширений: журнал пробитий, сценарии испытаний, пресеты и т. д.
4. Визуальная концепция

Визуально проект должен ощущаться как военный технический терминал.

Общий стиль:
темный фон;
светлые контрастные надписи;
аккуратные рамки;
минимализм без перегруза;
понятное деление на секции;
акцент на функциональность, а не украшения.
Впечатление от интерфейса:

Игрок должен чувствовать, что это:

не “чит-меню”;
не временная отладочная панель;
а штатный инструмент полигона.
5. Цветовая схема

Основной стиль предлагается такой:

Базовые цвета:
Фон: темно-серый / графитовый
Панели: чуть светлее фона
Текст: белый или светло-серый
Акцент: темно-синий / голубой / приглушенный бирюзовый
Предупреждения: желтый
Ошибки / опасные действия: красный
Подтверждение / успех: зеленый
Почему так:
на темном фоне лучше читается информация;
акцентные синие оттенки хорошо подходят под военный технический стиль;
интерфейс не выглядит игрушечным.
6. Общая структура интерфейса

Главное окно проекта должно состоять из пяти основных зон.

Верхняя панель

Содержит:

название проекта;
текущий режим;
кнопку закрытия;
кнопку обновления каталога.
Левая колонка

Это зона фильтрации:

сторона;
фракция;
нация;
тип техники;
поиск.
Центральная область

Это список техники:

найденные единицы;
сортировка;
прокрутка списка;
быстрый выбор.
Правая колонка

Это карточка выбранной техники:

название;
classname;
изображение;
фракция;
нация;
тип;
экипаж.
Нижняя панель

Это блок действий:

Spawn;
Spawn Empty;
Spawn With Crew;
Cleanup Range;
Spawn Target;
Tracking Settings.
7. Логика пользовательского опыта

Игрок должен проходить по очень простой цепочке:

Открыл полигон → нашел технику → увидел карточку → выбрал параметры → заспавнил → протестировал → очистил полигон

Это и есть основной UX проекта.

8. Экран каталога техники

Каталог — это главное сердце проекта.

Он должен быть удобным не только технически, но и визуально.
Игрок не должен видеть просто бесполезный массив classnames.
Он должен видеть понятную структуру.

Уровни фильтрации:
Сторона
Фракция
Нация
Тип техники
Примеры типов:
Tank
IFV
APC
Car
Truck
Static
Helicopter
Plane
Boat
UAV
Support
Что должно быть видно в списке:
название техники;
classname;
краткий тип;
фракция.
9. Карточка выбранной техники

Это одна из самых важных частей дизайна.

Когда игрок выбирает технику, справа должна появляться ее полноценная карточка.

Карточка должна содержать:
Название
Classname
Изображение
Сторону
Фракцию
Нацию
Тип
Класс экипажа
Основная задача карточки:

Игрок должен сразу понять:

что это за техника;
как она называется;
из какого она мода/фракции;
подходит ли она ему;
стоит ли ее спавнить.
10. Визуальный вид техники

Для первой версии проекта визуальный вид должен работать по простой и надежной схеме:

Приоритет:
editorPreview
picture
заглушка

То есть сначала пытаемся показать красивое превью техники, а если его нет — используем стандартную картинку.

Заглушка нужна для случаев:
у техники нет изображения;
путь сломан;
класс не содержит нормального preview.
11. Дизайн кнопок и действий

Все действия должны быть максимально понятными.

Основные кнопки:
Spawn — обычный спавн
Spawn Empty — пустая техника
Spawn With Crew — техника с экипажем
Cleanup Range — очистка полигона
Spawn Static Target — статичная мишень
Spawn Infantry Target — пехотная цель
Tracking Settings — настройки слежки
Логика цвета:
обычные действия — нейтральные;
важные — акцентные;
удаление и очистка — красноватые или предупреждающие.
12. Дизайн системы слежки за пулей/снарядом

Это отдельная важная визуальная часть проекта.

Система слежки должна быть:

красивой;
понятной;
не перегруженной;
не мешающей самому тесту.
Варианты режимов:
Simple
Tactical
Cinematic
Что можно показывать в HUD:
тип боеприпаса;
время полета;
дистанцию;
текущий режим tracking.
Общая визуальная идея tracking:

Это не должно быть хаотичное дерганое старое наблюдение.
Камера должна ощущаться как современный инструмент анализа выстрела.

13. Дизайн доступа к проекту

Открытие полигона должно происходить через:

горячую клавишу;
ACE self action;
терминал на объекте.
Почему это удобно:
для администратора — быстрый вход;
для сервера — можно ограничить доступ;
для сценариев — можно открыть через объект.
14. Внутренняя архитектурная идея

Проект должен ощущаться цельным, но внутри быть разбит на модули.

Модули:
mkk_ptg_core
mkk_ptg_ui
mkk_ptg_catalog
mkk_ptg_spawn
mkk_ptg_targets
mkk_ptg_tracking
mkk_ptg_ace
mkk_ptg_common
Зачем это нужно:

Такой подход позволит:

спокойно дорабатывать проект;
не ломать ядро при добавлении функций;
проще отлаживать;
быстрее находить ошибки.
15. Эмоциональное ощущение от проекта

Проект должен создавать ощущение:

аккуратности;
контроля;
технической надежности;
профессионального инструмента.

Игрок не должен чувствовать, что работает с “кучей старых скриптов”.
Он должен чувствовать, что у него в руках единый современный полигонный комплекс.

16. Что особенно важно сохранить в дальнейшем

При дальнейшей разработке нельзя терять три вещи:

1. Чистоту интерфейса

Не превращать окно в свалку функций.

2. Понятность каталога

Техника должна всегда находиться легко.

3. Стабильность сети

Спавн и удаление только по правильной серверной схеме.
