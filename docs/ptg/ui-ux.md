# MKK Proving Ground — требования к UI и UX

Использовать при изменении диалогов, компоновка, карточек, изображений, кнопок или общего пользовательского flow.

6. Общая структура интерфейса

Главное окно открывается как стартовая панель функций.

Стартовый экран содержит:

название проекта;
заголовок текущего раздела;
кнопку “Создание техники”;
кнопку “Камера”;
кнопку “Телепорт”;
кнопку “Слежение за пулей”;
блок “Быстрый маркер на карте”;
настройку цвета метки высоты;
блок “Тайминги на карте”;
кнопку “Создать взрыв”;
кнопку “Линия траектории пули”;
компактную кнопку настроек линии траектории;
кнопку “Маркеры снарядов на карте”;
компактную кнопку настроек маркеров снарядов;
кнопку “Статус-дисплей объектов”;
компактную кнопку настроек статус-дисплея;
кнопку “Бесконечные патроны”;
кнопку “Режим бога”;
кнопку “Инспектор hitpoints”;
компактную кнопку настроек инспектора hitpoints;
кнопку “Перевооружение”;
кнопку “Цели”;
комбо “Размер интерфейса”;
кнопку “Очистить полигон”;
кнопку “Разблокировать технику”;
кнопку “Закрыть”.

Функциональные кнопки стартового экрана располагаются ровной симметричной сеткой. Функции с настройками используют широкую основную кнопку и узкую icon-кнопку рядом с ней.

Левая колонка стартового экрана содержит табло действия кнопок: при наведении или фокусе кнопки в табло показывается ее локализованная подсказка. Центральная колонка содержит компактный список важных CBA-клавиш; список и подписи dashboard-кнопок с keybinds берут текущие пользовательские назначения из CBA.

Каталог техники скрыт при первом открытии окна.
Он появляется только после нажатия “Создание техники”.

7. Логика пользовательского опыта

Основной UX:

Открыл полигон → выбрал функцию → выполнил действие → вернулся назад или закрыл окно.

Для техники:

Открыл полигон → Создание техники → нашел технику → увидел карточку → задал дистанцию и угол → заспавнил → протестировал → очистил полигон.

Такой подход сохраняет главное окно чистым и не заставляет игрока каждый раз видеть каталог, если ему нужны функции вроде камеры, телепорта, быстрого маркера на карте, таймингов на карте, tracking, линии траектории, map markers, статус-дисплея, инспектора hitpoints, перевооружения, целей, режимов игрока или создания взрыва по карте. Камера и телепорт запускаются физическими кнопками в левом столбце dashboard или настраиваемыми CBA-клавишами.

Нижняя зона колонки создания содержит компактный блок “Быстрый маркер на карте”. Блок задает цвет дымовой подсветки; сам маркер ставится на открытой карте через настраиваемую CBA-клавишу (по умолчанию `Q`) под курсором и создает глобальный marker с именем игрока и smoke shell выбранного цвета. Marker удаляется через 3 секунды.

В этой же нижней зоне находится настройка цвета локальных меток высоты. Сами метки высоты всегда активны и ставятся только через настраиваемую CBA-клавишу на открытой карте.

Нижняя зона колонки создания содержит компактный блок “Тайминги на карте”. Блок содержит команды “Старт/Стоп”, “Очистить тайминги” и шестеренку настроек. Настройки управляют цветом локальных timing-маркеров, шагом постановки меток и опциональной скоростью в подписи marker. Нажатие “Старт/Стоп” при выключенном замере закрывает browser-dashboard, запускает фоновый локальный замер движения игрока и сразу ставит стартовый marker; повторное нажатие завершает замер с финальной меткой; “Очистить тайминги” удаляет только локальные timing-маркеры текущего клиента. То же переключение доступно через настраиваемую CBA-клавишу “Старт/Стоп таймингов” (по умолчанию `Ctrl+X`).

Актуальные дополнительные окна настроек стартового экрана:

линия траектории — цвет Cyan/Red/Yellow/Green/Magenta/White и толщина 1/2/3/5/8;
тайминги на карте — цвет marker, шаг 5/10/15/20/30/60 секунд и чекбокс скорости в подписи;
маркеры снарядов на карте — чекбокс “Класс ammo на маркере”;
статус-дисплей объектов — class, distance, total damage, AllowDamage, hitpoints и отдельные поля hull/engine/fuel/turret/gun;
инспектор hitpoints — выбор отображаемых категорий engine/hull/turret/gun/wheels/tracks/fuel tank и опциональное отображение 3D-точек/radius.

Основной operational UI рисуется локальной HTML-страницей в контроле `RscWebBrowser` (`type = 106`). Dashboard, каталог, цели, перевооружение и настройки используют единую визуальную систему. Дополнительные настройки открываются как модальная панель поверх dashboard; выбранные параметры применяются без закрытия панели. Browser view передает в SQF только команды пользователя, а спавн, удаление, перевооружение и игровое состояние остаются в существующих SQF-потоках.

Комбо размера интерфейса должно показывать “Малый/Обычный/Большой/Очень большой” через stringtable и сохранять выбранный множитель в `profileNamespace`. При смене размера должны перерасчитываться элементы главного окна, оверлеи и подсказка свободной камеры.

8. Экран создания техники

Экран создания техники состоит из пяти зон.

Верхняя зона:

название проекта;
текущий раздел “Создание техники”;
счетчик найденных объектов.

Левая колонка:

поиск;
сторона;
фракция;
тип техники;
совместимый ящик БК для выбранной статики;
дистанция спавна в метрах;
угол разворота в градусах.

Центральная область:

список найденной техники;
быстрый выбор;
прокрутка списка.

Правая колонка:

изображение техники;
карточка выбранной техники.

Нижняя панель:

создать пустую технику;
создать полный экипаж;
создать с экипажем;
очистить полигон;
разблокировать технику;
копировать classname выбранной техники;
назад.

Кнопки нижней панели должны быть выровнены ровно и использовать единый размер, чтобы экран не выглядел как набор временных отладочных элементов.

Кнопка “Создать полный экипаж” использует штатный экипаж выбранной техники и предназначена для пассивного спавна цели или объекта проверки. Кнопка “Создать с экипажем” создает водителя классом текущего игрока, сажает игрока в доступный слот стрелка и включает движение техники из этого слота через штатные Arma 3 actions движения, включая “полный газ”.

12. Карточка выбранной техники

Карточка должна показывать:

название;
classname;
изображение;
сторону;
фракцию;
тип;
группу / editorSubcategory;
класс экипажа;
источник мода.

Все подписи карточки должны быть локализованы через stringtable.

Основная задача карточки:

дать игроку быстро понять, что это за техника, из какой она фракции/группы, из какого источника пришла и стоит ли ее спавнить.

13. Визуальный вид техники

Для первой версии визуальный вид работает по простой и надежной схеме:

editorPreview;
picture;
пустое значение или будущая заглушка.

Если у техники нет изображения или путь сломан, интерфейс не должен ломаться.

14. Свободная камера

Настраиваемая CBA-клавиша “Камера” (по умолчанию K) запускает клиентский режим выбора точки на карте.

Ожидаемый flow:

нажатие клавиши K по умолчанию закрывает главное окно;
открывается карта с подсказкой выбрать область;
ЛКМ по карте сохраняет позицию и создает свободную камеру над выбранной точкой;
камера управляется WASD/Q/Z, мышью, колесом мыши и Shift;
при прокрутке колесика временно показывается текущая скорость камеры;
N включает или отключает ночное видение камеры;
F1 показывает или скрывает подсказку управления;
ЛКМ в режиме камеры переносит камеру в выбранную область;
настраиваемая клавиша “Закрыть камеру” (по умолчанию F) останавливает режим и очищает runtime-состояние.

Подсказка управления должна масштабироваться вместе с выбранным размером интерфейса и использовать только строки `STR_MKK_PTG_CAMERA*`.

14.1. Измерение высоты точки

Режим меток высоты активен постоянно и не имеет отдельной кнопки или статуса включения в dashboard. Через dashboard меняется только цвет marker в нижнем блоке картографических инструментов. Browser-dashboard не открывает карту сам: игрок открывает обычную карту, наводит курсор и нажимает настраиваемую CBA-клавишу “Высота точки” (по умолчанию `H`). Под курсором создается локальный маркер с одним коротким значением высоты без пробелов и дополнительных подписей. Созданные метки остаются на карте и не удаляются при закрытии карты.

15. Оверлей перевооружения

Кнопка “Перевооружение” открывает оверлей над главным меню только если игрок находится в технике.

Оверлей должен показывать:

название текущей техники;
изображение техники;
позиции экипажа / turret paths / driver weapons;
пилоны Arma 3 как отдельные строки выбора;
оружие выбранной позиции или текущий loadout пилона;
совместимые магазины из `CfgWeapons` или `CfgMagazines` для пилона;
информационную карточку выбранного магазина;
кнопку загрузки магазина;
кнопку очистки выбранного оружия/пилона;
кнопку копирования classname магазина;
кнопку обновления и закрытия оверлей.

Для обычных turret weapons источник совместимых магазинов — конфиг выбранного оружия. Список текущих магазинов техники не должен быть источником истины для совместимости.

Для пилонов нужно использовать конфиг `Components >> TransportPylonsComponent >> Pylons` и применять выбранный loadout через `setPylonLoadout` на машине, где техника local. Если пилон пустой или совместимых магазинов нет, UI должен показывать локализованный резервный вариант.

16. Оверлей тестовых целей

Кнопка “Цели” открывает оверлей создания тестовых целей.

Оверлей должен поддерживать:

режим боты-цели;
наземные цели;
воздушные цели;
настройки радиуса патруля от точки спавна для наземных целей, радиуса и высоты для воздушных целей;
предпросмотр выбранной цели;
создание цели через серверный request;
удаление созданных целей через серверный request.

Все подписи и сообщения оверлей должны идти через stringtable.

17. Окно создания взрыва

Кнопка “Создать взрыв” открывает отдельный `MKK_PTG_ExplosionDisplay` с browser-sidebar и нативной картой.

Окно должно поддерживать:

поиск по `CfgAmmo`;
категории bombs, rockets, ATGM/rockets и mortars через локализованные `STR_MKK_PTG_EXPLOSION_GROUP_*`;
выбор высоты создания projectile;
информационную карточку выбранного ammo: category, hit, indirectHit, indirectHitRange, typicalSpeed, submunitionAmmo и triggerDistance;
создание projectile по ЛКМ на карте и отправку его вниз;
проверку высоты против `triggerDistance` для ammo с `submunitionAmmo`;
восстановление выбранного ammo, поиска, категории и высоты после завершения tracking-камеры.

HTML/sidebar отвечает только за отображение и ввод. Нативная карта и SQF-обработчики остаются источником игровых действий.
