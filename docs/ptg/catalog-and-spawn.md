# MKK Proving Ground — каталог, фильтры и серверный спавн

Использовать при изменении поиска техники, фильтров, совместимых ящиков БК, логики спавна, реестра или очистки.

9. Каталог техники

Каталог строится динамически из CfgVehicles.

Не допускается жестко прописывать конкретные фракции из отдельных модов.
Проект должен корректно видеть технику из кастомных модов, если она зарегистрирована в конфиге Arma 3.

Текущая реализация `fnc_buildVehicleCatalog` добавляет только классы с `scope >= 2`, пропускает simulation `soldier`, `thing`, `fire`, `house` и берет только `LandVehicle`, `Air`, `Ship` и `StaticWeapon`.

Для каждой единицы техники сохраняются:

classname;
displayName техники;
сторона;
classname фракции;
displayName фракции;
ключ типа техники из stringtable;
класс экипажа;
editorPreview;
picture;
editorSubcategory;
displayName группы / подкатегории;
источник мода;
config entry.

Фактический порядок полей в записи каталога:

0 `classname`;
1 `displayName`;
2 `side`;
3 `faction`;
4 `vehicleType` как `STR_MKK_PTG_TYPE_*`;
5 `crewClass`;
6 `editorPreview`;
7 `picture`;
8 `editorSubcategory`;
9 `modSource`;
10 `cfg`;
11 `factionDisplayName`;
12 `editorSubcategoryDisplayName`.

Названия техники, фракций и групп должны проходить через локализацию:

если displayName содержит обычный текст, он используется напрямую;
если displayName содержит $STR_..., он должен быть преобразован через localize;
если значение отсутствует, используется безопасный резервный вариант.

10. Фильтрация каталога

Фильтры должны работать по стабильным техническим значениям, а отображать локализованные названия.

Например:

фильтр фракции хранит classname фракции;
в UI показывается displayName фракции;
фильтр типа хранит ключ `STR_MKK_PTG_TYPE_*`;
в UI показывается локализованное название типа.

Такой подход нужен, чтобы фильтрация не ломалась при смене языка игры.

Уровни фильтрации:

сторона;
фракция;
тип техники;
поиск по названию и classname. Поиск регистронезависимый и должен корректно работать с кириллицей: например, `бмп-2` находит технику с названием `БМП-2`.

11. Типы техники

Типы техники определяются автоматически по наследованию classname. В каталоге хранится не текст, а stringtable key `STR_MKK_PTG_TYPE_*`.

Текущий порядок проверки:

`Tank`;
`Wheeled_APC_F`;
`Tracked_APC_F`;
`Car`;
`Truck_F`;
`StaticWeapon`;
`Helicopter`;
`Plane`;
`Ship_F`;
`UAV_01_base_F`, `UAV_06_base_F`, `UAV`;
резервный вариант `Other`.

В UI типы должны отображаться через stringtable:

Tank → Танк;
IFV → БМП;
APC → БТР;
Car → Автомобиль;
Truck → Грузовик;
Static → Статичное;
Helicopter → Вертолет;
Plane → Самолет;
Boat → Лодка;
UAV → БПЛА;
Other → Другое.

12. Карточка выбранной техники

Карточка должна показывать:

название;
classname;
изображение;
сторону;
фракцию;
тип;
группу / editorSubcategory;
класс экипажа;
источник мода.

Все подписи карточки должны быть локализованы через stringtable.

Основная задача карточки:

дать игроку быстро понять, что это за техника, из какой она фракции/группы, из какого источника пришла и стоит ли ее спавнить.

13. Визуальный вид техники

Для первой версии визуальный вид работает по простой и надежной схеме:

editorPreview;
picture;
пустое значение или будущая заглушка.

Если у техники нет изображения или путь сломан, интерфейс не должен ломаться.

14. Спавн техники

Спавн техники должен выполняться по серверной схеме.

Клиент:

выбирает classname;
задает дистанцию;
задает угол разворота;
выбирает пустой спавн или спавн с экипажем;
при выборе статики может выбрать совместимый ящик БК;
отправляет запрос на сервер.

Сервер:

проверяет входные параметры;
создает технику на заданной дистанции перед игроком;
поворачивает технику с учетом угла разворота;
при необходимости создает экипаж;
для техники с экипажем обнуляет боезапас и топливо, чтобы экипаж не стрелял по игроку и не уезжал;
для пустой техники боезапас и топливо не меняются;
если выбрана статика и совместимый ящик БК, создает ящик рядом со статикой;
регистрирует созданный объект в реестре полигона.

Клиентский запрос проверяет доступ через `ptg_main_fnc_isAuthorized`, ограничивает дистанцию диапазоном от 1 до `mkk_ptg_spawnMaxDistance`, нормализует угол через `% 360` и отправляет `remoteExecCall` на сервер.

Текущие дефолты:

`mkk_ptg_spawnDefaultDistance` = 30;
`mkk_ptg_spawnMaxDistance` = 250.

Совместимые ящики БК для статики определяются динамически:

берутся weapons выбранной статики;
рекурсивно просматриваются Turrets;
по ним собираются применяемые magazines;
в CfgVehicles ищутся `ReammoBox_F` / `ReammoBox` классы со `scope >= 2`;
совпадение ищется в `TransportMagazines`;
в список показываются только подходящие ящики с displayName, source и списком совпавших magazines;
результаты кэшируются в `mkk_ptg_staticAmmoBoxCache`.

15. Очистка полигона

Очистка должна удалять:

созданную технику;
прочие зарегистрированные объекты полигона.

Удаление выполняется сервером.

Реестр очистки хранится в `mkk_ptg_spawnedVehicles` и `mkk_ptg_spawnedObjects`.
